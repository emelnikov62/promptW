# Сохранение лица: face-verify + авто-ретрай (Уровень C)

> Спека фичи «проверка сходства лица после генерации + тихий best-of авто-ретрай».
> Реализует **Уровень C** из `docs/template-authoring-and-face-fidelity.md` §6.
> Цель: довести узнаваемость лица в template-фотогенерациях с ~70–87% до ~99%.

Дата: 2026-06-22 · Статус: design approved, готова к плану реализации.

---

## 1. Проблема и цель

NanoBanana PRO стохастична (нет seed/strength) → даже с промт-слоем
(`_augment_template_prompt`, Уровень A) на проде ~13% явных промахов по личности
(мужчины ~80% узнаваемы > женщины ~60%). Промтом потолок ~90–93%.

Единственный способ **гарантировать** ~99% — пост-проверка результата по сходству лица
и тихий перегенерат при промахе. Математика (§6): база промаха ~13% → 1 ретрай ≈1.7%
→ 2 ретрая ≈0.2%.

## 2. Решения брейншторма (зафиксировано)

| Развилка | Решение |
|---|---|
| Движок сходства | **Локальный эмбеддинг InsightFace `buffalo_l` (ArcFace)** на VPS, косинус. RAM хватает. |
| Ретраи | **Тихо, бесплатно для юзера, до 2 ретраев** (best-of-3). |
| Финальный промах | **Лучший из попыток + мягкий совет** в тоне бренда. |
| Охват | **Только шаблоны с лицом** (`settings.tplId` + `photo-refs`) — тот же гейт, что у `_augment_template_prompt`. |
| Мультифото | В шаблонах всегда 1 выходная картинка → петля только для `count==1`. |
| Градация юзеру | **Не показывать** score; только внутри (логи/БД для калибровки). |

## 3. Архитектура

### 3.1 Новый модуль `face_verify.py` (изолированный, по образцу `storage.py`)

- Ленивая загрузка модели InsightFace `buffalo_l` один раз при старте (синглтон).
- `embed(image_bytes) -> list[float] | None` — детект лица + ArcFace-эмбеддинг
  (нормализованный). `None`, если лицо не найдено. При нескольких лицах — **самое крупное**
  (max bbox area).
- `similarity(a, b) -> float` — косинус нормализованных эмбеддингов.
- Все синхронные вызовы обёрнуты в `run_in_executor` (async-обёртки `a*`), чтобы не
  блокировать aiohttp-loop (как `storage.a*`).
- **Опциональный импорт**: `insightface`/`onnxruntime` отсутствуют или модель не грузится
  → модуль репортит `available() == False`; генерация идёт штатно (**fail-open**, фича
  никогда не ломает основной поток).
- Модель — в **gitignored** `FACE_MODEL_ROOT` (напр. `/opt/tg-image-ai-bot/models/`), НЕ в
  git-чекаут (иначе ломается `git pull --ff-only` на деплое).

### 3.2 Точка встраивания

Логика — **внутри `_build()`** в `api/routes.py::generate_image` (детач-таск из
`_run_generation`), поэтому переживает дисконнект клиента, а строка финализируется один
раз — уже выбранным лучшим кадром.

Новый хелпер `_generate_with_face_verify(effective_prompt, model, settings, files, gen_id, on_task)`
оборачивает существующий вызов `generator.generate_image(...)`.

## 4. Поток данных (best-of петля)

1. **Гейт** (иначе — текущий одиночный путь без изменений):
   `FACE_VERIFY=1` (или shadow, см. §8) И `face_verify.available()` И `settings.tplId`
   И `files["photo-refs"]` И `settings.get("count", 1) == 1` (count может отсутствовать —
   дефолт 1, как в route) И генератор = KIE (не Stub).
2. **Эмбеддинг референса — один раз**: `files["photo-refs"]` (в s3-режиме это уже публичные
   S3-URL после `_publish_refs`) → `storage.aget_bytes` → `embed`. Несколько рефов — берём
   первый с валидным лицом. **Нет лица в рефе → фолбэк на обычную одиночную генерацию**
   (логируем, `face_ref_found=false`, не блокируем).
3. **Петля попыток** `attempt = 1 .. 1+MAX_RETRIES`:
   - перед каждой новой попыткой проверяем **тайм-бюджет** (`FACE_VERIFY_BUDGET_SEC`); не
     укладываемся → выходим с лучшим, что есть;
   - `generator.generate_image(...)` (тот же `effective_prompt`; повторяемость даёт чистая
     стохастика) → `result`, `paths`;
   - результат → `storage.aget_bytes`/чтение локального файла → `embed` → `similarity`;
   - кладём `(score, result, paths)` в кандидаты; пишем score в `face_scores`;
   - `score >= T_accept` → принимаем, выход;
4. **Выбор**: кандидат с максимальным `score`. Медиа **непринятых** кандидатов удаляем через
   `_discard_media` (нет orphan-объектов в S3).
5. **Финализация**: `provider_task_ids` схлопываем до id принятой попытки (см. §5), затем
   `finish_generation_if_pending(best_url, best_urls)` как сейчас. Пишем
   `face_attempts`/`face_score`/`face_accepted`/`face_ref_found`/`face_scores`/`face_threshold`.
6. `best_score < T_accept` → в ответ добавляем `face_tip` (см. §7).

### Граничные случаи
- Лица нет в результате попытки → `score = -1.0` (= провал, идёт ретрай). Если так во всех —
  отдаём первый (max по score, т.е. «наименее плохой»), `face_accepted=false`.
- Любая ошибка `face_verify` (битый файл, исключение) → ловим, трактуем как «не смогли
  проверить» → отдаём текущий результат без ретрая (**fail-open**).
- `count > 1` (на будущее, если шаблон начнёт отдавать N) → гейт не срабатывает, штатный путь.

## 5. Согласование с reconciler и тайм-бюджет

**Риск:** петля делает несколько последовательных KIE-вызовов. Reconciler рефандит строку,
которая `pending` и «processing > 30 мин» (`_RECONCILE_GIVEUP=1800`). NanoBanana под нагрузкой
бывает >300с/вызов → 3 попытки могут подобраться к лимиту.

**Рычаг 1 — жёсткий тайм-бюджет** `FACE_VERIFY_BUDGET_SEC` (деф. 900с): не начинаем новую
попытку, если не укладываемся; выходим с лучшим. Гарантирует, что best-of не уводит строку
за `GIVEUP`.

**Рычаг 2 — `provider_task_ids` по выбранному кандидату**: `on_task`/`_task_saver` по ходу
петли накапливает id всех попыток (это ок для recovery), а на финализации список схлопываем
до id принятой попытки (новый хелпер в `queries.py`, напр. `set_generation_task_ids(gen_id, ids)`).
Тогда при рестарте сервиса прямо в петле reconciler восстановит **валидный** кадр (пусть и не
лучший по similarity), а не отвергнутый. Допустимая деградация: при краше юзер получает
нормальное лицо без best-of-гарантии.

## 6. Персистентность + статистика в админке

### 6.1 Колонки `generations` (идемпотентная миграция в `database.py`, все nullable)
- `face_attempts SMALLINT` — число KIE-вызовов (1/2/3).
- `face_score REAL` — финальный лучший косинус.
- `face_accepted BOOLEAN` — прошёл ли best порог.
- `face_ref_found BOOLEAN` — было ли лицо в референсе (проверка реально применялась).
- `face_scores JSONB` — массив скоров по попыткам (постфактум-калибровка порога).
- `face_threshold REAL` — порог на момент генерации (env тюнится → иначе исторические данные «плывут»).

`tplId` (в `settings` JSONB) / `model` / `cost` / `created_at` уже в строке → разрезы по
шаблону и периоду без новых таблиц.

### 6.2 Деньги (потери на ретраях)
Юзеру списываем 1 раз (30 ток.), KIE-кредиты тратим на каждую попытку.
**Потери = `Σ(face_attempts − 1) × FACE_VERIFY_RETRY_UNIT_COST`** (₽ за один прогон, env).
Считаем в дашборде по текущей цене (исторический стамп цены — оверкилл, не делаем).

### 6.3 Запросы (`queries.py`)
- `get_face_verify_stats(period)` — агрегаты за день/неделю/месяц/всё (паттерн `ref_earnings`).
- `get_face_verify_by_template(period)` — разрез по шаблонам (`settings->>'tplId'`).

### 6.4 Admin API
`GET /api/admin/face-stats?period=` — под `admin_scope` (как остальной админ; `_qint`-парсеры).

### 6.5 Дашборд («Сходство лиц» в `admin.html`/`admin.js`)
- Всего eligible-генераций; % где проверка реально шла (`face_ref_found`).
- Распределение попыток 1/2/3 (кол-во + %), retry-rate (% где ≥2).
- Acceptance-rate: с 1-й попытки и итоговый после ретраев.
- Средний/медианный best-score + гистограмма по бэндам (отлично/ок/слабо) — калибровка порога.
- **Потери на ретраях: всего лишних прогонов и ₽** за период.
- **Таблица по шаблонам**: топ по retry-rate / низкому score — сигнал, какой шаблон чинить промтом.
- Фильтр периода (день/неделя/месяц/всё).

## 7. UX «мягкий совет»

При `best_score < T_accept` ответ `generate_image` несёт необязательное `face_tip: <i18n-key>`
(сам score НЕ отдаём). Фронт рисует ненавязчивую плашку под результатом в тоне бренда
(«лучший друг»), максимум один раз на результат:
- ru: «Хотел(а) бы похожесть получше? Загрузи фото анфас при хорошем свете — и магия точнее ✨»
- en/es — паритет ключей в `i18n.js`.

Никаких «сходство 47%».

## 8. Конфиг (env) и план раскатки

Все дефолты — фича **выключена**:
- `FACE_VERIFY=0` — мастер-флаг (ретраи).
- `FACE_VERIFY_SHADOW=0` — теневой режим: считать score и писать в БД, но **без** ретраев
  (сбор калибровочных данных без трат).
- `FACE_VERIFY_THRESHOLD=0.35` — порог принятия (откалибровать по `face_scores`).
- `FACE_VERIFY_MAX_RETRIES=2` — итого до 3 прогонов.
- `FACE_VERIFY_BUDGET_SEC=900` — тайм-бюджет петли.
- `FACE_MODEL_ROOT=/opt/tg-image-ai-bot/models` — путь модели (gitignored).
- `FACE_VERIFY_RETRY_UNIT_COST=<₽>` — стоимость одного прогона для подсчёта потерь.

**Поэтапная раскатка:**
1. Деплой кода + `pip install insightface onnxruntime` + предзагрузка `buffalo_l` в
   `FACE_MODEL_ROOT`; `FACE_VERIFY=0`, `FACE_VERIFY_SHADOW=1` (собираем реальные score без
   трат на ретраи).
2. Смотрим распределение score в админке → выставляем `FACE_VERIFY_THRESHOLD`.
3. `FACE_VERIFY=1` (петля включена), следим за retry-rate и потерями в дашборде.

## 10. Вне охвата (YAGNI / на будущее)

- Уровень B (мульти-референс UX-наджадж, авто-кроп лица, гейт качества загрузки) — отдельная спека.
- Face-verify для свободной страницы «Создать» (без `tplId`) — пока нет.
- Параллельный best-of-N — отвергли (всегда тратит N× кредитов).
- Точный исторический стамп цены прогона — оверкилл.
- Bytes результата для эмбеддинга качаются повторно (после `_download_file`); микро-оптимизация
  переиспользования локальных байтов — не сейчас.
