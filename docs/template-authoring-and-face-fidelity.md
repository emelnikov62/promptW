# PromptW — Создание шаблонов и сохранение лица (полный гайд)

> Самодостаточный документ. Можно вставить в новый чат целиком — в нём весь
> накопленный опыт по шаблонам («трендам») и по тому, как добиться, чтобы
> сгенерированный человек был тем же, что на фото-референсе. Канонический
> проектный гайд — `bot-src/CLAUDE.md`; этот файл — углублённое дополнение по
> шаблонам и идентичности.

---

## 0. TL;DR для нетерпеливых

При создании нового фото-шаблона с лицом:
1. **Промт описывает ТОЛЬКО сцену/одежду/свет/ракурс. Никогда — внешность человека.**
   Внешность целиком берётся из фото. Любое словесное описание лица конкурирует
   с фото и тянет результат к «обобщённому красивому незнакомцу».
2. **Никаких слов-праймингов**: «успешный / уверенный / статусный / эффектный /
   гламурный / супермодель / VOGUE / editorial / высокая мода / красивый». Они
   провоцируют подмену на «модель». (Часть из них и так вырезается кодом — см. §5,
   но не закладывай их сознательно.)
3. **Кадр — поясной или средний план, лицо крупно.** Не «в полный рост» — мелкое
   лицо = модель достраивает черты сама = чужой человек.
4. **Поля загрузки:** фото-шаблон — `photo-refs`, видео-шаблон — `v-first-frame`
   (НЕ `photo`).
5. Всё, что в §0 п.1–3, частично **дожимается автоматически** скрытым кодом-слоем
   `_augment_template_prompt` (де-прайминг + английский identity-блок + перекадровка).
   Но не полагайся только на него — пиши промт сразу правильно.

## 1. Где живут шаблоны (архитектура данных)

- **Данные шаблонов — в БД**, таблица `templates` (с PR #46), НЕ инлайн в `app.js`.
  - Лёгкие колонки: `id / type / enabled / sort_order / category / cost / title /
    preview / featured / need_photo`.
  - JSONB `definition`: `model / settings / skeleton / prompt / params / desc / …`.
- **Сид при старте** из `db/templates_seed.json` через `_seed_templates` с
  **`ON CONFLICT (id) DO NOTHING`** → деплой только ДОБАВЛЯет новые шаблоны,
  **никогда не перезаписывает** правки, сделанные в админке. Строкой владеет админка.
- **Движок параметров — клиентский** (`app.js`: `buildTplPrompt` / `renderTplParams`
  / `tplGenerate`). `var TRENDS = {}` — ленивый кэш: `showTplDetail(id)` →
  `loadTemplateThen(id)` → `GET /api/templates/{id}`. Лёгкий список — `GET /api/templates`.
- **Цена авторитетна на сервере и плоская за шаблон** — `pricing.TEMPLATE_COST`
  загружается из `templates.cost` при старте, обновляется при каждой админ-правке.
  Цена редактируется в админке/БД, НЕ в коде.

### Создание нового шаблона — 100% из админки, без деплоя
Браузерная админка (`admin.html`/`admin.js` + `admin_routes.py`,
`/api/admin/templates` GET/POST/PUT/DELETE):
1. Создать строку: `id`, `type` (`photo`/`video`), `cost`, `sort_order`, `category`
   (`girls`/`kids`/`men` — порядок/лейблы из `TPL_CAT_ORDER`), `enabled`, `featured`
   (bool → попадает в верхнюю сетку «Тренды»).
2. Заголовки RU/EN/ES.
3. Превью: загрузить файл (`POST /api/admin/templates/upload`, multipart, ≤30 МБ,
   ext-allowlist → S3/`/media`) → запишется в `preview.img`/`full`. Или путь
   `/static/tpl/<id>.jpg` (закоммить файл; **не писать в git-чекаут в рантайме** —
   ломает `git pull --ff-only` на деплое).
4. «Настройки модели»: model / ratio / ratios / quality / duration / mode /
   minPhotos / maxPhotos / **refField** + тогглы sound / needPhoto / hidePrompt.
5. «Промпт»: skeleton + готовый prompt.
6. «Описание» RU/EN/ES.
7. `params` — визуальный редактор (см. §7), JSON руками не нужен.

`saveTemplate` пересобирает `definition` из исходного объекта (незнакомые ключи
выживают) и выкидывает пустые скаляры/false-боолы.

## 2. КРИТИЧНО: фото-референс должен дойти до модели под правильным ключом

Два слоя, оба ломались исторически:

**(1) Клиентское поле формы:**
- фото-шаблоны грузят референс под **`photo-refs`**,
- видео-шаблоны — под **`v-first-frame`**,
- **НЕ `photo`** (молча теряет референс). Выбирается по `type` в `tplGenerate`.

**(2) Серверный ключ для KIE — per-model** (был дорогой баг):
- **NanoBanana PRO** (фото, `nano-banana-pro`): читает `image_input` — **массив**
  (мультиреференс поддержан!) через `_collect_files(files, "photo-refs")`.
- **Seedance 2.0** (`bytedance/seedance-2`, fast `…-2-fast`): `first_frame_url` (str)
  + `reference_image_urls` (array, резолвит `@Image1` в промте); `duration` — int;
  аудио — `generate_audio`; `resolution` 480p/720p.
- **Grok Imagine 1.5**: `image_urls` (array, ≤7); `duration` — **строка** (6–30).
- **Kling 3.0**: `image_urls` (array) + `tail_image_url` (str).
- **Kling Motion 3.0**: `subject_reference` + `video` + `character_orientation`.
- ⚠️ Для нового видео-шаблона **всегда** сверяй точную input-схему на docs.kie.ai —
  не предполагай `image`. Старый `kie.generate_video` клал картинку в `"image"`,
  модели его игнорировали → text-to-video → случайное (часто азиатское) лицо.

⚠️ У NanoBanana PRO **нет seed/strength** — генерация полностью стохастична.
Поэтому повторяемость даётся только промтом, входом (референсы) и пост-проверкой.

## 3. Главные причины «не тот человек» (диагностировано на проде)

Три корневые причины дрейфа личности, в порядке убывания важности:

1. **Размер лица в кадре (главное).** Полноростовые/широкие кадры → лицо занимает
   мало пикселей → модель достраивает черты из текстового prior'а промта → выдаёт
   обобщённого «модельного» незнакомца. Прод-A/B: supercar «в полный рост» 3/4
   (1 чужой) → поясной 4/4; alpine после перекадрировки 4/4; jet (поясной) и так 4/4.
   С фиксом — 12/12 узнаваемо. **Решение: поясной/средний план, лицо крупно.**
2. **Слова-прайминги внешности.** «успешный/уверенный/статусный мужчина»,
   «эстетика супермодели», «VOGUE/editorial/высокая мода» → модель подменяет лицо
   на аспирационную «модель» (у мужчин характерный тэлл — **дорисованная борода**;
   у женщин — моложе/худее/«инстаграм-модель»). **Решение: убрать прайминг.**
3. **Оверсайз-референс молча выкидывается.** Фулл-рез фото с телефона
   (3024×4032 / 6 МБ) заставляет KIE/NanoBanana **проигнорировать `image_input`**
   целиком → лицо из промта (как будто референс не отправился). Тот же селфи в
   960×1280 применяется 1:1. **Решение уже в коде** (`_shrink_image` ≤1536px по
   длинной стороне, вызывается в `_parse_request` для каждой картинки — покрывает
   ген-референсы, сохранённые рефы, видео-кадры). Если лица вдруг перестали
   переноситься — первым делом проверь размеры/байты референса.

НЕ являются причиной (проверено): даунскейл до 1536, русский язык промта.

## 4. Правила написания промта нового шаблона (для сохранения лица)

### 4.1 Внешность — только из фото, никогда из текста
- ❌ НЕ хардкодь черты: причёску/«фейд», бороду/«щетину», этничность/«славянской
  внешности», возраст, телосложение, цвет волос/глаз как факт.
- ✅ Вместо этого нейтральная отсылка к референсу:
  «мужчина (внешность, причёска, длина и тип волос, наличие или отсутствие бороды и
  усов, телосложение — строго как на загруженном фото-референсе, ничего не добавлять
  и не убирать)».
- Почему: когда превью делают по одному тест-лицу, авторы запекают черты этого лица
  в сцену, и она ПЕРЕБИВАЕТ фото у любого непохожего юзера (длинные волосы /
  гладко выбрит и т.п.).

### 4.2 Front-load identity-блока (в начало промта)
Все боевые промты начинаются мощным блоком (пример из боевого `man-alpine`):
> «КЛЮЧЕВОЕ ТРЕБОВАНИЕ: точно перенести лицо с загруженного фото-референса, это
> портрет КОНКРЕТНОГО человека, а не собирательный образ… Скопируй лицо 1:1:
> сохрани без изменения форму и ШИРИНУ лица и полноту щёк (НЕ сужай, не утончай, не
> вытягивай овал), линию челюсти, форму и пропорции носа, форму и разрез глаз,
> расстояние между глазами, брови, форму губ, скулы, текстуру кожи, родинки,
> щетину/бороду и её форму, причёску — 100% узнаваемость. ЦВЕТ ГЛАЗ — строго как на
> референсе. ВАЖНО: это обычный человек, а НЕ фотомодель — НЕ заменяй лицо на типовое
> «модельное», не делай худее/скуластее/моложе/симметричнее. БЕЗ сглаживания кожи,
> БЕЗ AI-эффекта, не идеализируй, не омолаживай. Если сомневаешься — копируй точно.»

Бери этот блок за основу для новых шаблонов (адаптируй пол).

### 4.3 Защитные отрицания — оставляй слово «модель» ТОЛЬКО в них
Фраза «НЕ заменяй на типовое «модельное»» полезна. НЕ пиши «модель» в позитивном
ключе («как супермодель», «модельная внешность»).

### 4.4 Кадр и поза
- Явно проси **поясной/средний план**, лицо крупно и в фокусе. Не «в полный рост».
- **Пин позы**, если сцена двусмысленна. Пример (yacht): «стоит НА палубе, спиной
  опирается на перила, перила ЗА спиной, НЕ висит снаружи/сквозь перила» — иначе
  модель выносит человека за борт над водой.

### 4.5 Стилизация (Ч/Б, editorial) — самый рискованный случай
В чёрно-белом/высокой моде лицо «фэшнится» сильнее всего (girl-vogue был 1/3 —
слабейший шаблон). Если без стилизации не обойтись — добавь явное:
«Чёрно-белая/editorial-стилизация НЕ меняет черты лица — это то же лицо с референса,
просто в монохроме; сохрани реальные пропорции и ширину лица». (Код добавляет
английский аналог автоматически при обнаружении монохром-кук — см. §5.)

### 4.6 Убери «улучшайзеры» сцены
Прилагательные вроде «роскошное», «интенсивный взгляд», «эффектный» подталкивают к
идеализации. Описывай свет/фактуры/окружение нейтрально-технически (объектив,
освещение, материалы), а не оценочно про человека.

## 5. Что код делает автоматически (`api/routes.py::_augment_template_prompt`)

Применяется к КАЖДОЙ template-фотогенерации, у которой есть `settings.tplId` И
загружен `photo-refs`. Свободная страница «Создать» (без `tplId`) и генерации без
референса НЕ трогаются. Хранимый/возвращаемый промт остаётся ЧИСТЫМ (слой
применяется на лету каждый раз; промты от юзеров спрятаны в UI). Без правки прод-БД.

Конвейер (по порядку):
1. **Детект стилизации** на исходном тексте (`_BW_CUES`: чёрно-бел/ч-б/монохром/
   grayscale/black and white/editorial).
2. **Перекадровка** (`_FULLBODY_REPL`): «кадр в полный рост по грудь» → «поясной
   портрет, крупный план по грудь»; «в полный рост» → «поясной/средний план».
3. **Де-прайминг** (`_DEPRIME_REPL`, regex, IGNORECASE) — вырезает ПОЗИТИВНЫЙ
   прайминг: `успешн / уверенн / статусн / эффектн / гламурн`, `роскошн +(человек/
   мужчин/женщин/девушк/парн)`, `супермодел`, `модел +(внешност/лиц)`,
   `красив / привлекательн / симпатичн / обаятельн / очаровательн`, и фэшн-куки
   `VOGUE / editorial / высокая мода / fashion-съёмка`. ⚠️ Бэр «модель» НЕ трогает
   (живёт в защитных отрицаниях). Чистка двойных пробелов/висячих запятых.
4. **Английский primacy-якорь в начало** (`_TPL_IDENTITY_ANCHOR`): «используй
   референс как строгий источник лица; описание ниже — только сцена/одежда/свет/
   кадр, не внешность».
5. **Английский identity-суффикс в конец** (`_TPL_IDENTITY_SUFFIX`, recency):
   «тот же реальный сфотографированный человек, НЕ модель/двойник/знаменитость,
   копируй лицо 1:1; не улучшай/не сужай/не меняй возраст; НЕ добавляй бороду/усы,
   которых нет (и не убирай, если есть); кадр крупный, лицо в фокусе». На английском
   намеренно — модель на базе Gemini точнее слушает EN.
6. **Условный Ч/Б-блок** (`_BW_CLAUSE`) — добавляется только если был детект из п.1.

> История: де-прайминг + перекадровка — PR #57; английский блок + якорь + Ч/Б-блок +
> точечный де-прайминг — PR #60. Грубое правило `модель→человек` было откатано: оно
> портило защитные отрицания в боевых промтах.

**Практический вывод для автора шаблона:** пиши промт правильно сам (§4), а
код-слой — страховка, а не замена. Не полагайся на него для удаления того, что
можно просто не писать.

## 6. Дополнительные рычаги к ~99% (стратегия)

Честно: **только промтом 99% не взять** — модель стохастична (нет seed/strength),
потолок промт-правок ≈ 90–93%. Замер на проде (30 генераций, 10 шаблонов × 3, оба
референса): ~70% уверенных / ~13% явных промахов (мужчины ~80% > женщины ~60%;
лучшие — man-alpine/gangster 3/3, худший — girl-vogue 1/3).

- **Уровень A — промт (СДЕЛАНО, §4–5).** Де-прайминг, английский identity-блок,
  перекадровка, анти-борода/анти-возраст, Ч/Б-блок. → ~90–93%.
- **Уровень B — вход (не сделано, снижает базовый промах):**
  1. **Мультиреференс** — UX-наджадж «загрузи 2–3 фото с разных ракурсов» (бэкенд
     уже шлёт `image_input` массивом). Самый сильный рычаг базовой точности.
  2. **Авто-кроп лица** — на сервере детектить лицо и добавлять плотный кроп
     отдельным элементом `image_input` (густой identity-сигнал независимо от кадра).
  3. **Гейт качества загрузки** — предупреждать ДО генерации, если лицо мелкое/в
     профиль/в очках/смазано.
- **Уровень C — реальный закрыватель до 99% (не сделано):**
  **Face-similarity verify + тихий авто-ретрай.** Считать косинус эмбеддингов лица
  (ArcFace/InsightFace) результат↔референс; если ниже порога — молча перегенерить
  1–2 раза до показа. Математика: база промаха ~13% → 1 ретрай ≈1.7% → 2 ретрая
  ≈0.2%. С мультиреференсом база падает ещё, ретраи дешевеют. Это единственный
  способ ГАРАНТИРОВАТЬ ~99%.

## 7. Система параметров (parameterized prompts)

Шаблон может прятать сырой промт и показывать кураторские параметры
(пол/возраст/одежда/причёска), из которых промт собирается детерминированно.

- `skeleton` — фикс-проза со слотами `{subject}/{hair}/{outfit}/{age}` + кастомными
  (`girl-roses` использует `{flowers}/{slippers}/{watch}/{phone}`). Identity-предложение
  ВСЕГДА первое и фиксированное.
- `params` — массив `{id, control:"pills"|"sheet"|"age", label:{ru,en,es}, default,
  dependsOn?, options}`. Каждая опция: `label:{ru,en,es}` (UI) + `frag` (русский
  фрагмент промта). Пол несёт `grammar:{noun,adjE}`; одежда/причёска **gender-keyed**
  (`options:{female:[...],male:[...]}` + `dependsOn:"gender"`).
- `buildTplPrompt(tpl, params)` — чистая детерминированная интерполяция, всегда
  непустая (дефолты). `tplGenerate` пишет `settings.tplId` + `settings.tplParams`.
- Наличие `params` переключает UI деталей с textarea на контролы.
- **Female-only**: опусти `gender` целиком → движок пропустит subject/gender-логику.
- **«Без значения»** опция: дай реальный frag («без наручных часов»), не пустой
  `frag:""` — иначе в промте остаётся «. .».
- `buildTplPrompt` вплетает «~N лет» в `{subject}` только если есть `age`-param
  (age-less шаблоны получают чистый noun).
- Редактор `params` в админке полностью визуальный (PR #54): кнопки `+ Пол/+
  Возраст/+ Список/+ Список по полу`, правки мутируют клон in-place (без потери
  фокуса), добавление/удаление — ре-рендер. Движок (`initTplParamState`/`tplFirstVal`)
  падает на первую опцию, если нет `default`.

## 8. Превью, размещение, цена

- **Превью**: генерь из первого хорошего прогона (скачай результат в
  `webapp/static/tpl/<id>.jpg`) ИЛИ грузи через upload-эндпоинт в админке. Не писать
  в git-чекаут в рантайме. Прод-storage = S3 (превью персистят off-box).
- Самогенерация превью через прод NanoBanana PRO возможна (см.
  «promptw-preview-pipeline»); URL референсов — S3-объекты, KIE их забирает.
- **Размещение**: дом-галерея динамическая (`renderTemplatesHome()` из
  `GET /api/templates`). Категории из `category`; клиентский `TPL_CAT_ORDER =
  ["girls","kids","men"]`. `featured` → верхняя сетка «Тренды». `[data-tpl]` клики
  и «все →» — делегированные. Мобайл-готча: `.tpl-thumb { flex:0 0 110px; width:110px }`
  (иначе `<img width:100%>` раздувается до натурального размера в iOS webview).
- **Цена** редактируется в админке/БД (`templates.cost`), не в коде.
- Все 10 текущих шаблонов — NanoBanana PRO, 9:16 (+3:4), params:true,
  minPhotos 1 / maxPhotos 8, 30 токенов.

## 9. Деплой и совместная работа (важно для безопасности изменений)

- Репо = `bot-src/`, ветка `main`, GitHub `emelnikov62/promptW`. VPS = git-чекаут
  `origin/main`, `/opt/tg-image-ai-bot/`, сервис `promptw`, домен promptw.ru,
  nginx→localhost:8081.
- **Единственный метод деплоя**: ветка → PR → `gh pr merge` → на машине владельца
  `bash srv2.sh gitdeploy` (git pull --ff-only + pip-если-изменился + restart).
  Никогда не мешать с rsync, никогда не пушить прямо в main.
- **Правки промтов/идентичности — в КОДЕ** (`_augment_template_prompt`), НЕ прямым
  UPDATE по shared prod-таблице `templates` (заблокировано и правильно — строкой
  владеет админка). Контент-правки конкретного шаблона — через админку.
- ⚠️ Часто несколько сессий делят рабочий каталог. Чтобы не смешать чужие
  незакоммиченные правки со своими — лендить через отдельный
  `git worktree add -b <branch> /tmp/land origin/main`, коммитить там только свои
  файлы, мёржить, потом `git worktree remove`. Проверено на PR #57/#60.
- Бэкенд-правки промтов не требуют бампа `?v=N` ассетов и рестарта статики.
- ⚠️ Не деплой без нужды, пока рендерится видео (sweep восстановит за ~60с, но это
  лишняя churn). Координируй рестарты.
